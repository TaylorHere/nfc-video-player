# NFC Video Player - 系统架构与数据流文档

本文档详细描述了当前 NFC 防伪视频播放系统的架构、各端职责及完整的数据处理流程。

## 1. 系统架构概览

本系统由三个核心部分组成：
1.  **写入端 (PC / Writer)**：负责初始化 NTAG 424 DNA 卡片，配置加密密钥，并开启 SUN (Secure Unique NFC) 防伪功能。
2.  **读取端 (Mobile App)**：用户终端。负责读取 NFC 卡片生成的动态链接，并将其发送给服务器进行验证。
3.  **网络端 (Server / Backend)**：安全中心。负责持有解密密钥，验证链接的真实性，并返回对应的视频内容。

---

## 2. 各端详细职责与数据流

### 2.1 写入端 (PC / Writer)
*   **软件**：`CallNtag424mainwindow.py` (基于 Python/PyQt5 + OUR_MIFARE.dll) 或自定义的 `nfc-writer`。
*   **硬件**：ACR122U 或其他支持 ISO 7816-4 的读写器。
*   **职责**：
    1.  **认证**：使用 Master Key (Key 0) 获取卡片控制权。
    2.  **配置密钥**：将 Key 1-4 更改为生产密钥 (e.g., `5189...`)。
    3.  **配置 SUN**：开启 SDM (Secure Dynamic Messaging) 功能，设置镜像偏移量 (Mirror Offsets)。
    4.  **写入 URL**：写入带有占位符的 NDEF 链接。
        *   *写入格式*：`http://<ServerIP>:8000/verify?p=0000...&m=0000...`
*   **输出结果**：一张经过特殊配置的 NFC 卡片。卡片每次被刷时，芯片会自动用 Key 1 加密 UID 和计数器填充到 `p` 参数，用 Key 2 计算签名填充到 `m` 参数。

### 2.2 读取端 (Mobile App)
*   **软件**：Flutter App (`flutter_app`)。
*   **职责**：
    1.  **感应 (Tap)**：用户将手机靠近卡片。
    2.  **获取动态 URL**：手机 NFC 模块直接读取卡片生成的完整 URL。
        *   *读取到的信息*：`http://192.168.1.5:8000/verify?p=E2A8...&m=5F3C...` (注意：`p` 和 `m` 每次都是变化的乱码)。
    3.  **转发**：App 识别到这是一个验证链接，将其通过 HTTP GET 请求发送给后端服务器。
    4.  **播放**：接收服务器返回的视频流或重定向地址，播放视频。

### 2.3 网络端 (Server / Backend)
*   **软件**：FastAPI 后端 (`main.py`)。
*   **核心配置**：
    *   `SDM_KEY` = `5189...` (必须与写入端设置的 Key 1/2 一致)。
*   **处理流程**：
    1.  **接收**：收到 `GET /verify?p=...&m=...` 请求。
    2.  **验证签名 (CMAC)**：使用 `SDM_KEY` 对 URL 参数进行签名计算，对比 `m` 值。如果对不上，说明链接被篡改，拒绝访问。
    3.  **解密 (Decrypt)**：使用 `SDM_KEY` 对 `p` 值进行 AES-128 解密。
        *   *解密出的信息*：**真实 UID** (如 `04 82 5A...`) 和 **读取计数器** (如 `5`)。
    4.  **业务逻辑**：
        *   查询数据库中该 UID 对应的视频 URL。
        *   (可选) 检查计数器是否重复（防重放）。
    5.  **响应**：
        *   如果验证通过：返回 `302 Redirect` 跳转到视频文件 (e.g., `butterfly.mp4`)。
        *   如果验证失败：返回错误信息。

---

## 3. 完整业务流程图 (Sequence)

1.  **[用户]** 刷卡 -> **[卡片芯片]** 内部计算加密数据 -> 生成动态 URL。
2.  **[手机 App]** 读取到动态 URL -> 发送 HTTP 请求给 **[服务器]**。
3.  **[服务器]** 用密钥解密 URL -> 拿到真实 UID -> 查库找到视频 -> 返回视频地址。
4.  **[手机 App]** 收到视频地址 -> 开始播放。

## 4. 安全性总结
*   **卡片**：无法被克隆（因为克隆卡没有芯片内部的密钥，无法生成正确的 `m` 签名）。
*   **链接**：无法被分享（因为链接是一次性的，包含计数器，服务器可拒绝旧链接）。
*   **内容**：只有持有真卡的人，通过官方服务器验证后，才能看到视频。
